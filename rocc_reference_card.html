Rule engine design
Rules stored in DecisionRule table with priority ranking.
Rule engine loads on startup, caches with configurable TTL.
Priority-based evaluation: first matching rule wins.
Admins update rules via ROLE_ADMIN API — no redeployment needed.
Every evaluation logged: ruleId · hazardId · action · durationMs

AffectedTrainResolver — 4-step algorithm
1
Subdivision filter
SELECT trains WHERE subdivision_id = hazard.subdivision_id AND status = ACTIVE
2
Spatial proximity filter (PostGIS)
ST_DWithin(train.location, hazard.location, radius_meters)
Uses spatial index — O(log n) not O(n). Default impact radius: 5 miles.
3
Direction filter — safety-critical
bearing = ST_Azimuth(train.location → hazard.location)
Include if: ABS(angleDiff(bearing, train.direction)) ≤ 90°
Exclude if: train heading AWAY from hazard
Stale direction data (>60s) → INCLUDE train. Fail-safe: over-notify rather than miss.
4
Mile post filter (belt-and-suspenders)
If mile post data available: exclude trains already PAST hazard mile post in direction of travel.
Unit test scenarios
✓ Approaching train — must be included
✗ Receding train — must be excluded
⚠ Perpendicular train — excluded (bearing diff >90°)
⚠ Stale direction — must be included (fail-safe)
✓ Multi-subdivision — other subdivisions excluded
✓ Beyond mile post — already-passed trains excluded
Fail-safe scenarios — safety always wins
Decision Engine down
Circuit opens → Hazard Service auto-issues PROCEED_WITH_CAUTION → logs degraded-mode event → alerts ROLE_ADMIN
Train Tracking down
AffectedTrainResolver falls back to full subdivision broadcast — all active trains notified, no filtering applied
Kafka down
Producers write to local_event_store (DB table) → FailsafeEventScheduler polls every 30s → replays in order with deduplication
Notification failure
Retry 3× with exponential backoff → DLQ after exhaustion → DLQ consumer triggers email/SMS backup → dispatcher sees UNDELIVERED badge
WebSocket lost
Client reconnects with backoff (1s→2s→4s, max 30s) → server buffers notifications during disconnect → flush buffer on reconnect before new ones
Safety principle
Safety always takes precedence over throughput. When in doubt, the system errs toward restriction (PROCEED_WITH_CAUTION or higher) rather than letting trains operate normally under uncertainty. This aligns with FRA and GCOR operating principles.

Likely questions from Ms. Rabi & Mr. Hale — with answer anchors
1
"Why event-driven for a safety system? Doesn't async introduce risk?"
At-least-once delivery + Kafka's durable log means no event is lost. Synchronous REST would couple services — if Decision Engine is slow, hazard reporting blocks. Async decoupling lets each service fail independently with defined fallbacks. The 2s budget is satisfied with 1.6s margin.
2
"What happens if the Decision Engine goes down mid-operation?"
Circuit breaker opens after 5 failures in a 10-call window. Hazard Reporting's FallbackFactory auto-issues PROCEED_WITH_CAUTION — trains slow down, no danger. Event is buffered in local_event_store. When Decision Engine recovers, the event replays and a proper decision is generated. Dispatcher is alerted to degraded mode.
3
"How does AffectedTrainResolver know which trains to notify?"
4-step algorithm: (1) subdivision filter, (2) PostGIS ST_DWithin spatial proximity, (3) direction-bearing filter — only trains whose heading is within ±90° of the bearing toward the hazard, (4) mile post filter to exclude trains already past the hazard. Stale direction data triggers fail-safe inclusion.
4
"How does this scale to Union Pacific's 1,700+ locomotives?"
Kafka partitioned by subdivisionId — each partition handled by one Decision Engine instance. K8s HPA scales Train Communication pods on WebSocket connection count. PostGIS spatial index (GiST) keeps ST_DWithin queries at O(log n) regardless of fleet size. Stateless services scale horizontally; only DB needs vertical scaling or read replicas.
5
"How do you ensure a train never misses a critical notification?"
Three layers: (1) Kafka at-least-once with @RetryableTopic — 3 retry attempts before DLQ. (2) WebSocket offline buffering — notifications queued per trainId during disconnect, flushed on reconnect. (3) DLQ consumer triggers backup alerting (email/SMS). Every notification has ACK tracking — dispatcher sees UNDELIVERED status in real time.
6
"What FRA / GCOR compliance does this address?"
Dedicated Audit & Compliance Service with append-only AuditEvent table — no updates, no deletes. Every action has actor_id, trace_id, and timestamp. Decision rules stored in DB with effective_from timestamps (historical rule state recoverable). Notification ACKs provide proof of delivery. TraceId propagates end-to-end through HTTP → Kafka → WebSocket for complete audit trail.
7
"What would you build next with more time?"
PTC (Positive Train Control) integration via ATCS protocol adapters. ML-based severity prediction using historical hazard + outcome data. Geospatial subdivision boundary management from actual UPRR track topology. Real-time weather API integration for predictive hazard scoring. Mobile app for locomotive engineer ACK (currently WebSocket-only).
